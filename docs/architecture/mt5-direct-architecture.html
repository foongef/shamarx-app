Designation

MT5 DIRECT

Self-hosted MT5 bridge — our own MetaApi, on our own metal. One Windows host runs a fleet of portable MT5 terminals; the existing Broker abstraction treats it as just another backend. Zero third-party bridge. Zero per-account fees. Friends onboard through the same wizard.

◆ Section · 01 Constraints first

Design goals

GoalHow the design answers it
Save money nowPhase 1 runs on one t3.small Windows box (~$37/mo all-in) serving 3–4 accounts. No new services, no load balancers, no MetaApi fees for migrated accounts.
No stability sacrificeWatchdog supervises every terminal + worker with exponential-backoff restarts; heartbeats feed the same loop-health pill the dashboard already shows; nightly EBS snapshot = 15-min full rebuild.
No security sacrificeThe Windows host has zero public ingress — admin via SSM only (same as the app server), API reachable only from the app server's security group over the private VPC. Passwords AES-256-GCM at rest, never logged, never leave the VPC.
Automated onboardingUser pastes login/password/server in the wizard → terminal provisioned, logged in, health-checked, and trading in ~60 seconds. No human touches the host.
Scalable laterHost capacity is a column, not an assumption. Phase 2 adds hosts behind a registry table; Phase 3 bakes a golden AMI so new hosts self-provision via Terraform. The API contract never changes.

Components

terminal-manager
Single FastAPI service on the Windows host. Owns the terminal fleet: provisions from a golden portable-MT5 template, spawns one worker per terminal, routes per-account Broker calls (/t/:id/orders, /t/:id/positions, …) to the right worker, reports capacity.
worker (×N)
One Python process per terminal — the official MetaTrader5 lib binds process-globally, so isolation is mandatory, not optional. Implements the same verbs as every other Broker: order, positions, modify, close, account-info, candles.
watchdog
30-second supervision loop: terminal process alive? worker IPC responding? broker connection state OK? Restart with exponential backoff (1m→2m→4m→cap 15m), write per-terminal heartbeat to Redis — surfaces in the existing loop-health pill automatically.
Mt5DirectClient
~150 lines in execution-service implementing the Broker ABC by HTTP-calling the manager. The registry dispatches on broker='MT5_DIRECT' exactly like CTRADER/METAAPI. Nothing upstream changes — engine, risk, analytics, journal all blind to the swap.
Mt5Host registry (DB)
New table: id, privateIp, capacity, activeCount, status. Phase 1 has one row. BrokerAccount gains hostId. Provisioning picks the least-loaded healthy host — the Phase 2 shard key already in place on day one.
golden template
A pristine portable MT5 folder (no login, charts off, news off, sounds off, minimal symbols) zipped on the host. Provisioning = unzip + write config.ini + launch. Versioned so terminal upgrades roll fleet-wide deliberately, not accidentally.
◆ Section · 03 User adds an account · fully automated

Onboarding flow

01
Wizard — same one that exists today
/accounts/new gains an “MT5 Direct” card next to cTrader. Form: login, password, server name (e.g. ICMarketsSC-Demo), friendly name. The custody disclosure is shown here, plainly: “Shamarx stores this password encrypted to operate your terminal.”
02
Backend: encrypt, persist, pick a host
Creds → AES-256-GCM (existing CryptoService) → BrokerAccount row (broker='MT5_DIRECT', isEnabled=false). Host selection: least-loaded healthy row in Mt5Host with free capacity; 409 with a clear message if the fleet is full.
03
Provision call — private VPC, shared-secret header
POST http://10.x.x.x:8100/terminals {accountId, login, password, server}. Manager unzips the golden template into terminals/<accountId>/, writes the login config, launches the terminal in portable mode.
04
Login verification — fail fast, fail clean
Manager polls terminal state up to 90s. Connected → spawn worker, register in process table, return healthy. Auth failed → kill terminal, delete folder, return 401 — wizard shows “check your credentials” and the BrokerAccount row is removed. No zombie terminals, no orphan rows.
05
Go live
Backend stamps lastConnectedAt, flips the account enabled (or honors the wizard’s “enable now” checkbox). Engine fans out to it at the next M15 close. Wall-clock for the user: ~60 seconds, zero human involvement.
06
Removal mirrors creation
Account delete in the UI → DELETE /terminals/:id → worker stopped, terminal killed, folder shredded, host capacity freed. Same open-trades guard the delete flow already enforces.
◆ Section · 04 Threat model honored

Security posture

SurfaceControl
Network ingressNo public IP on the Windows host. Security group allows :8100 from the app server’s SG only. RDP closed entirely — admin via SSM Session Manager (port-forward RDP through SSM on the rare day a GUI is needed), identical to how we manage the Linux box today.
API authShared-secret header (from AWS Secrets Manager, same pattern as shamarx/ctrader-oauth) on every manager call; manager rejects everything else. Upgrade path to mTLS in Phase 2 if friends’ LIVE accounts justify it.
Passwords at rest (app side)Existing AES-256-GCM CryptoService, key in Secrets Manager. Same custody class as MetaApi creds today.
Passwords at rest (host side)Written once into the terminal’s own config at provision; never logged, never echoed in any API response, EBS volume encrypted (default-on). Folder shredded on deprovision.
Blast radiusThe host can trade accounts but holds no Shamarx DB creds, no JWT secrets, no other users’ web data. Compromise is severe for trading accounts on that host — which is why ingress is zero and the box runs nothing else.
User consentMaster-password custody stated in the wizard in plain words before submission. Friends who are uncomfortable use the cTrader OAuth path instead — both rails coexist.
PatchingWeekly Windows Update window (Sun 22:00 UTC, market closed); watchdog quiesces and resumes the fleet around the reboot automatically.
Stated trade-off: unlike cTrader OAuth, MT5 has no delegated-auth mechanism — custody of the master password is intrinsic to ANY MT5 bridge (MetaApi holds it today too). Self-hosting doesn’t add this risk; it relocates it from MetaApi’s cloud to a box we control and can audit.
◆ Section · 05 Boring by design

Stability mechanics

FailureDetectionRecovery
Terminal crashwatchdog · 30sRelaunch from its folder, auto-login, worker reattaches. Backoff 1m→15m on flapping; alert pill after 3 strikes.
Worker hangIPC ping timeoutKill + respawn worker (terminal untouched — positions unaffected).
Broker disconnectterminal state pollMT5 reconnects natively; watchdog escalates only if disconnected > 5 min.
Whole host downRedis heartbeat TTLLoop-health pill goes DEGRADED (same UX as tonight’s MetaApi outage). SL/TP live on the broker server — open positions stay protected exactly as they are right now. Rebuild from nightly AMI snapshot ≈ 15 min.
Windows reboot (patches)scheduled windowWatchdog runs as a service with auto-start; fleet resumes unattended.
Capacity exhaustionmanager refuses > capProvisioning 409s with “fleet full” — never oversubscribes RAM into OOM territory. Adding capacity = Phase 2 playbook below.
The reconciliation hardening shipped this week (3-strike absence confirmation, error-skips-reconcile) was built for exactly this class of bridge flap — MT5 Direct inherits it for free.
◆ Section · 06 Money, honestly

Cost model

PhaseInfraCapacity$/month$/account
Phase 1 1× t3.small Windows + 50GB gp3 3–4 terminals ~$37 ~$9–12
Phase 2 upgrade in place → t3.large 8–12 terminals ~$85 ~$7–10
Phase 3 N hosts from golden AMI (Terraform) 10–12 per host ~$85 / host ~$7
Reference MetaApi G2 high-reliability per account — ~$30–50

Break-even vs MetaApi: ~1–2 accounts. Phase 1 → 2 is a stop-instance/resize/start (10 min, no code). Reserved-instance pricing shaves a further ~30% once committed. Ops time honestly budgeted: ~1–2 hrs/month steady-state.

◆ Section · 07 De-risked sequence

Build & rollout plan

W1
Host + manager + one terminal (manual account first) · ~4 days
Terraform the Windows EC2 (SSM-only, SG locked), golden template, terminal-manager with provision/route/destroy, worker, watchdog. Provision your own IC Markets demo by API call — not via UI yet.
W2
Wire into the platform · ~3 days
Mt5DirectClient + registry dispatch + Mt5Host table + wizard “MT5 Direct” branch + provisioning orchestration in NestJS. Candles from the worker feed the same Candle table — a second independent candle source for free.
W2–3
Parallel-run gate · 1–2 weeks, zero code
Your demo runs on BOTH MetaApi and MT5 Direct simultaneously (two BrokerAccounts, same MT5 login). Compare fills, candle parity, uptime. Only after parity: migrate the primary, demote MetaApi to fallback, open the wizard branch to friends.
∞
Endgame topology
cTrader OAuth the smooth zero-custody rail (pending approval) · MT5 Direct for MT5-only friends, self-hosted · MetaApi legacy fallback, retire when count hits zero. One Broker ABC over all three — no engine changes ever.
Total build estimate: ~7–9 days engineering + a 1–2-week passive parallel-run. Next step if this design is approved: formal spec → implementation plan (same Spec-1/2/3 process), starting with the W1 block.
